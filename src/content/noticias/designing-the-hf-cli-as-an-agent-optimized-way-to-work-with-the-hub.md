hf é o ponto de entrada oficial da linha de comando para o Hugging Face Hub. Qualquer coisa que você possa fazer no Hub a partir do Python SDK, você pode fazer a partir do seu terminal: baixar e carregar modelos, conjuntos de dados e Spaces; criar e gerenciar repositórios, ramificações, tags e pull requests; executar Jobs na infraestrutura de HF; gerenciar Buckets, Collections, webhooks e Inference Endpoints.

A CLI do hf foi construída principalmente para nossos usuários ao longo dos anos. Mas agora é cada vez mais usado por agentes de codificação: Claude Code, Codex, Cursor e muito mais. Então, nós o reconstruímos para que funcionasse para os dois públicos ao mesmo tempo. Este post resume o que fizemos e como o comparamos. Descobrimos que, em tarefas complexas e de várias etapas, a linha de base sem CLI (um curl de rolagem manual do agente ou o Python SDK) usa até 6× tantos tokens quanto o CLI do hf.

Começamos a rastrear o uso do Hub pelos agentes em abril de 2026. O hf CLI (e o huggingface_hub Python SDK em que é construído) detecta quando um agente de codificação o está conduzindo lendo o conjunto de agentes de variáveis de ambiente: CLAUDECODE/ CLAUDE_CODE para Claude Code, CODEX_SANDBOX para Codex, além de Cursor, Gemini, Pi e o universal AI_AGENT . Esse único sinal faz dois trabalhos: ele molda a saída da CLI (mais abaixo) e marca cada solicitação do Hub com um agente/&lt;name&gt; user-agent, para que possamos atribuir o tráfego ao agente que o conduz. Os dois maiores por usuários distintos são Claude Code e Codex , bem à frente de todo o resto, e eles são os dois agentes que comparamos mais adiante neste artigo.

As barras contam usuários distintos por agente; o volume de solicitação é o sub-etiqueta. O Claude Code sozinho tem cerca de 40 mil usuários e quase 49 MILHÕES de solicitações, com o Codex logo atrás. Estes são números iniciais (só começamos a atribuir tráfego de agentes em abril de 2026), mas a escala já é significativa e esperamos que continue crescendo à medida que os agentes de codificação se tornem uma maneira padrão de trabalhar com o Hub.

Humanos e agentes de codificação esperam saídas diferentes para os mesmos comandos hf. Um humano quer uma saída de terminal rica: cor ANSI, mesas acolchoadas truncadas para caber na tela, um verde ✅ no sucesso, ✔ para booleanos, barras de progresso, dicas de prosa. Um agente quer o inverso: sem ANSI, nada truncado, todos os valores na íntegra, uma vez que um agente pode lidar com uma saída muito mais densa do que um humano, mantido compacto e estruturado para fique leve nas fichas. Ele também não pode responder a um prompt da CLI e terá prazer em executar novamente um comando após um tempo limite. O resto desta seção é como o hf dá a cada lado o que ele precisa. Introduzimos a saída de modo de agente no hf v1.9.0 e migramos o restante da CLI para ele gradualmente nas versões seguintes.

Quando o hf detecta automaticamente o uso do agente (por meio das variáveis de ambiente mencionadas acima), ele processa o mesmo comando de maneira diferente. Otimiza o formato de saída para humanos ou agentes sem passar um sinalizador:

Um agente obtém o registro completo como TSV: IDs de repositório completos, carimbos de data e hora ISO completos, cada tag, sem códigos ANSI, nada truncado, limpo para analisar e iluminar os tokens.

Na prática, implementamos métodos de registro em log como .table(...) ,.result (...) , .json() , etc., que usam dados brutos como entrada e manipulam a formatação. Além dos modos humano e agente, introduzimos as opções --json e --quiet para facilitar o pipe de comandos juntos. O modo padrão é escolhido automaticamente com base no contexto, mas os usuários sempre podem forçar o formato de sua escolha com --format human | agent | json | quiet .
